# Daily Briefing（AI 新闻雷达）— 项目计划

> 基于 LangGraph + Tavily + LLM 的新一代 AI 日报 Agent，继承 `ai-news-briefing` 的简报格式与 Prompt 经验，目标是把产品做好：少而精、可读性强、可扩展。

---

## 1. 项目定位与目标

### 1.1 定位

- **是什么**：一个**独立的新项目**，用 LangGraph 编排的「Search → Filter → Summary → Save」图，自动生成每日 AI 简报 Markdown。
- **不做**：不修改现有 `ai/ai-news-briefing` 项目；该项目继续按现有方式运行，保持稳定交付。

### 1.2 目标产出

| 产出 | 说明 |
|------|------|
| 可运行的 LangGraph 工作流 | 输入「今日」触发，输出一份简报文件 |
| 信源接入可靠 | Search 节点通过 Tavily（+ 可选 RSS）稳定获取多条 AI 相关条目 |
| 端到端一键出报 | 运行 Agent 自动生成 `daily_news/YYYY-MM-DD_daily_news.md`，含 frontmatter + 摘要 + 三章节 |
| 与前端兼容且样式升级 | 输出格式保留 frontmatter + 摘要行 + 三章节；展示层按 2.2 原则（Notion 风、真实页面设计）做样式，不采用明显 Markdown 式排版 |

### 1.3 目标用户与使用场景

- **目标用户画像**：
  - **对 AI 感兴趣的工程师群体**：包括后端/前端/全栈工程师、正在转型的 AI 工程师，以及已经在做 RAG/Agent/Infra 的同学。
  - **AI 周边岗位**：对 AI 产品有强兴趣的产品经理、UX/UI/视觉设计师等，需要跟进行业但不想被堆满链接的「信息瀑布」冲击。
  - 共性：对技术细节与产品落地都关心，同时对**视觉与阅读体验有较高要求**，希望简报页面本身也有现代、克制的设计感。
- **主要使用场景**：
  - 每天早上或晚上，花 **10–15 分钟** 打开一页，**从头到尾读完** 当日简报，而不是只扫标题。
  - 在工作或项目讨论前，快速了解最近有哪些值得引用/讨论的 AI 事件、产品或模型进展。
- **内容与表现取舍原则**：
  - 内容上：宁可条数少一些，也要保证每条都有「背景 + 要点」，读完之后对该事件/产品有一个完整 mental model，而不是只知道标题。
  - 表现上：页面设计需符合现代产品/设计从业者的审美（简洁、科技感、类似 Notion / 高质量科技媒体），避免「粗糙 Markdown 渲染」感。

## 2. 从 ai-news-briefing 继承与改进

以下内容在新项目中**复用或迁移**，并在**内容充实度**与**页面样式**上按下方要求做改进，而非原样照搬。

### 2.1 简报内容与 Prompt（继承 + 内容充实）

- **摘要行**：正文第一行必须为「摘要：」开头，后跟 1～2 句话概括当日核心，用于列表/详情描述。
- **三章节结构**（Markdown）：
  1. **## 1. 本周值得关注的 AI 产品与工具**（3～5 条）
  2. **## 2. 各场景下的头部模型与玩家**（按场景归纳 2～4 个代表模型）
  3. **## 3. 今日 AI 大事件与重要言论**（3～5 条）
- **风格**：理性、科技感、偏极客；基于给定内容撰写，不编造；链接统一用 `[原文链接](URL)（来源：XXX）`。

**内容充实要求（相对 ai-news-briefing 的改进）**：

- 目标：**读者不点开链接也能理解这条新闻在讲什么、为什么重要**；避免「只有标题 + 一句话 + 链接」的单薄罗列。
- 每条入选内容需包含：
  - **背景介绍**：事件/产品/公司是什么、在什么语境下发生（例如行业背景、此前进展）。
  - **要点概括**：2～4 句实质性信息——关键数据、核心观点、与已有生态的关系、为何值得关注；可适当引用信源中的具体表述。
  - **延伸阅读**：保留原文链接作为「了解更多」入口，而非唯一信息源。

**Prompt 优化方向**（用于 Summary 节点）：

- 在 prompt 中**明确要求**：为每条条目撰写「背景 + 要点」，禁止只写标题和一句笼统描述就接链接。
- 可规定每条最少字数或句数（例如每条至少 2～4 句完整句，且需包含至少一处具体信息：数字、人名、产品名、时间点等）。
- 可要求模型「先提炼信源中的关键事实与背景，再组织成可独立阅读的段落」，避免照抄标题。
- 实现：在 `prompts/summary_prompt.txt`（或等价模板）中落实上述要求，仅基于 **Filter 后的条目** 生成简报；格式与气质可参考 `ai-learning-plan/knowledge_base/daily_ai_news_workflow.md` 的 30 分钟简报思路，结构沿用 ai-news-briefing 的三章节；可参考 ai-news-briefing 的 `llm_processor.py`，但**必须加入上述充实度约束**。

### 2.2 页面样式与展示原则（相对 ai-news-briefing 的改进）

- **问题**：当前 ai-news-briefing 的展示偏「明显 Markdown 布局」（# / ## / ### + 列表），阅读体验不够像正式新闻/编辑页。
- **目标风格**：
  - **参考**：美国权威新闻/科技媒体的阅读页与列表页，如 **The Verge、TechCrunch、MIT Technology Review、Reuters** 等——层次清晰、留白合理、信息密度与可读性平衡。
  - **气质**：类似 **Notion**——简洁、科技感、不花哨；**不要非常明显的 Markdown 布局**，而是**真正的页面设计**。
- **具体要求**：
  - 前端呈现上：通过**字体层级、卡片/区块化、间距与分隔、引用块或高亮**等，让页面像 editorial 文章页，而非「渲染后的原始 Markdown」。
  - 避免：一眼就是「一级标题 + 二级标题 + 无序列表」的堆砌；标题与正文、不同条目之间应有清晰的视觉层次和留白。
  - 实现层面：本 Agent 项目产出仍为 Markdown；可在 Markdown 中**保留语义化结构**（如固定章节、可选 shortcode 或约定区块标记），便于前端用组件或 class 做差异化样式；**展示层**（新前端或改造现有 ai-news-web）单独按上述原则做 UI 与排版设计。

### 2.3 输出格式与解析

- **文件命名**：**采用** `YYYY-MM-DD_daily_news.md`，输出目录为 `daily_news/`（若现有前端强依赖 `YYYY-MM-DD_AINews.md` 或 `daily_reports/`，可在 config 中通过 `OUTPUT_DIR` 与命名模板切换）。
- **YAML Frontmatter**：`title`、`date`、`description`、`slug`（建议 `slug` 用日期如 `YYYY-MM-DD` 或 `daily-YYYY-MM-DD`，便于 SSG/Next 路由）。
- **摘要解析**：若 LLM 输出首行为「摘要：xxx」，则解析出 `description` 写入 frontmatter，并从正文中去掉该行；逻辑可参考 ai-news-briefing 的 `dispatcher._parse_summary_and_content`（若该项目不在本仓库，则自实现：按首行匹配「摘要：」并拆分为 description + 正文）。

### 2.4 信源策略

- **多信源、带来源**：每条内容保留 `source`、`title`、`link`，可选 `summary`。
- **RSS 作为可选补充**：Tavily 为主信源；若需增强覆盖，可增加 1～2 个 RSS（如 Hacker News AI、机器之心），与 Tavily 结果合并去重后进入 Filter。

### 2.5 部署与自动化思路

- **单次运行**：本地或 CI 执行一次 Agent，生成当日简报文件。
- **与前端配合**：生成的文件可被拷贝到前端 content 目录（如 `ai-news-web/content/reports`），由 GitHub Actions 或本地脚本同步；展示层按 2.2 的样式原则做设计，本计划阶段可先实现「生成到本地目录」，前端样式在接入时统一优化。

## 3. 技术架构

### 3.1 整体流程（LangGraph 图）

```
[Start]
   ↓
Search Node（Tavily + 可选 RSS）
   ↓
raw_items（列表，含 source / title / link / snippet 等）
   ↓
Filter Node（LLM：过滤无关、广告、低质量）
   ↓
filtered_items
   ↓
Summary Node（LLM：按继承的 prompt 生成简报正文）
   ↓
report_markdown（含「摘要：」+ 三章节）
   ↓
Save Node（写文件 + frontmatter，可选 MCP）
   ↓
[End]
```

- **顺序图**：无分支循环，线性四节点即可满足当前目标；后续若加「重试」「人工确认」再引入条件边。
- **错误处理**：任一节点失败时可写入 `state.error` 并终止图；Save 节点可判断 `report_markdown` 是否为空再写文件，避免覆盖已有简报。

### 3.2 State 设计（建议）

用 `TypedDict` 或 Pydantic 定义「图状态」，在节点间传递，例如：

| 字段 | 类型 | 说明 |
|------|------|------|
| `raw_items` | `list[dict]` | Search 节点输出：原始条目（含 source, title, link, snippet 等） |
| `filtered_items` | `list[dict]` | Filter 节点输出：通过过滤的条目 |
| `report_markdown` | `str` | Summary 节点输出：简报正文（含「摘要：」行 + 三章节） |
| `report_path` | `str \| None` | Save 节点输出：最终文件路径 |
| `error` | `str \| None` | 任意节点出错时写入，便于终止或降级 |

- 若使用 LangGraph 的 `StateGraph`，每个节点接收全 state，返回「部分更新」（只返回要更新的字段）。

### 3.3 节点说明

| 节点 | 输入依赖 | 行为 | 输出更新 |
|------|----------|------|----------|
| **Search** | 无（或 config 中的关键词） | 调用 Tavily API，关键词如 "Top AI News today" / "AI Agents latest news" / "LLM trends 2026"；可选拉取 1～2 个 RSS，合并去重 | `raw_items` |
| **Filter** | `raw_items` | 将 raw_items 转成文本，调用 LLM 做「是否与 AI 相关、是否广告/噪音」判断，返回保留条目列表（或索引） | `filtered_items` |
| **Summary** | `filtered_items` | 使用继承自 ai-news-briefing 的 prompt 模板，仅基于 filtered_items 生成简报 Markdown | `report_markdown` |
| **Save** | `report_markdown` | 解析摘要行、生成 frontmatter、写入 `daily_news/YYYY-MM-DD_daily_news.md`（或 `daily_reports/`）；可选：通过 MCP 写文件 | `report_path` |

### 3.4 依赖与配置

- **Python**：3.10+。
- **依赖**：
  - `langgraph`、`langchain-core`、`langchain-openai`（或你已有的 LLM 封装）
  - Tavily：`tavily-python` 或 `requests` 调用 Tavily API
  - 可选：`feedparser`（若做 RSS 补充）、`python-dotenv`
- **配置项**（如 `.env` + `config.py`）：
  - `TAVILY_API_KEY`（必填，Tavily）
  - `LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL`（与 ai-news-briefing 一致，便于复用）
  - `OUTPUT_DIR`：输出目录，默认 `daily_news` 或 `daily_reports`
  - 可选：RSS 列表、Tavily 搜索关键词列表、Filter/Summary 的 model/temperature

## 6. 与 ai-news-briefing 的差异小结

| 维度 | ai-news-briefing | daily-briefing（本项目） |
|------|------------------|---------------------------|
| 信源 | 仅 RSS | Tavily 为主，RSS 可选 |
| 流程 | 线性：fetch → summarize → save | LangGraph：Search → Filter → Summary → Save |
| 过滤 | 无 | 有 Filter 节点（LLM） |
| 内容充实度 | 偏单薄，多标题+链接 | 每条含背景+要点，不点链接也能读懂（Prompt 优化） |
| 输出格式 | frontmatter + 三章节 | 继承，一致 |
| 页面样式 | 偏明显 Markdown 布局 | Notion 风、简洁科技感、真实页面设计（见 2.2） |
| 保存 | Python 直接写文件 | Python 写文件；可选 MCP |
| 产品差异点 | 无特别强调 | 少而精 + 工程师视角 + 结构化可扩展（见下） |

**本项目的 3 个核心差异点（产品视角）**：

1. **少而精**：每天控制在约 6–12 条条目，全部经过 Search + Filter 精选；相比「信息瀑布式」简报，更希望读者愿意完整读完一页，而不是只扫标题。
2. **工程师视角**：刻意强调「这对 RAG / Agent / Infra / 产品落地意味着什么」，而不是泛媒体的「AI 又要改变世界了」；选题和写作偏向能启发工程实践的内容。
3. **结构化 + 可扩展**：底层用 LangGraph 把 Search / Filter / Summary / Save 做成显式图，将来可以方便接多数据源、多工具、多 Agent（例如添加代码分析、仓库扫描、领域特定数据等），形成真正可演进的「AI 新闻雷达引擎」。

## 7. 产品验收标准

- **信源**：运行 Search 节点（或独立脚本），能通过 Tavily（+ 可选 RSS）稳定获取并输出多条最新 AI 相关条目的标题与链接。
- **端到端**：运行完整 Agent（main.py），能自动生成 **`daily_news/YYYY-MM-DD_daily_news.md`**（或 config 指定路径与命名），内容为带 frontmatter 的「摘要 + 三章节」简报，基于 Filter 后的结果，每条具备背景 + 要点、可独立阅读。
- **可选**：若需与 MCP 生态集成，Save 节点可接入 Filesystem MCP；否则 Python 直接写文件即可。
